# ADR-0001 기획(PRD/ERD) 대비 실제 구현 변경점

## 상태

승인됨

## 배경

초기 해커톤 기획 단계에서 작성된 PRD(v1) 및 ERD를 바탕으로 개발을 진행하던 중, 실제 서비스의 운영 가능성과 사용자 경험(UX)을 끌어올리기 위해서는 초기 스펙을 넘어서는 확장 기능(소셜 로그인, 관리자 기능, 보안 강화 등)이 필수적이라는 판단을 내렸습니다. 또한 공공데이터 API의 실제 응답 스펙과 기획 문서 간의 불일치 현상이 발생하여, 데이터 파이프라인의 설계와 비즈니스 로직을 애자일(Agile)하게 즉각 수정할 필요가 있었습니다.

## 결정

기획안(PRD/ERD)의 기본 틀을 유지하되, 서비스 운영 가능성과 UX 극대화를 위해 다음과 같이 5가지 영역에서 아키텍처 및 로직을 고도화/변경하기로 결정했습니다. 상세 내역은 아래와 같습니다.

### 1. 회원 인증 및 권한 관리 (Auth & Users)
* **소셜 로그인(OAuth) 확장 도입**
  * **기획(PRD):** 이메일/비밀번호 기반의 자체 회원가입 및 JWT 인증만 설계
  * **실제 구현:** 사용자 편의성 극대화를 위해 **Google 및 Kakao 소셜 로그인** 추가 구현
  * **DB 변경점:** `users` 테이블에 소셜 연동을 위한 `auth_provider` 및 `oauth_id` 컬럼 추가
* **관리자(Admin) 권한 및 계정 상태 관리 추가**
  * **기획(PRD):** 일반 사용자(User) 중심의 서비스 로직만 존재
  * **실제 구현:** 최고 관리자 개념 도입 및 악성 유저 차단 기능 추가
  * **DB 변경점:** `users` 테이블에 `role` (ADMIN, USER, COMPANY) 및 `status` (active, blocked) 컬럼 추가

### 2. 기업 증빙 및 서류 관리 (Company Verification)
* **기업 인증 절차 고도화**
  * **기획(PRD):** `COMPANY_PROFILES`에 기본 정보만 저장
  * **실제 구현:** 가입 시 증빙 서류를 제출하고, 관리자가 대시보드에서 이를 확인한 후 **승인(APPROVED) 또는 반려(REJECTED)** 처리하는 프로세스 구현
  * **DB 변경점:** `company_profiles` 테이블에 `is_verified` 및 `verification_status` 컬럼 신설

### 3. 관리자 대시보드 (Admin Dashboard) 신설
* **기획(PRD):** 서비스 사용자를 위한 프론트엔드 뷰만 존재
* **실제 구현:** 실시간 운영 현황 파악을 위한 강력한 관리자 전용 React 대시보드 전면 구축
  1. **실시간 통계 모니터링:** 누적 가입자, 활성 조건, 수집된 공고, 메일 발송 현황 시각화
  2. **수동 배치 실행:** 관리자가 즉시 공고 수집 배치를 실행할 수 있는 기능
  3. **통합 회원 관리:** 가입된 모든 회원의 상태 변경 및 관리자 임명/해임, 삭제 기능

### 4. 데이터베이스 및 스크래핑(배치) 로직 최적화
* **마감일(Deadline) 파싱 오류 수정**
  * **발생 원인:** 나라장터 OpenAPI 문서의 마감일 키값을 `bidEndDt`로 예상했으나 실제는 `bidClseDt`로 반환됨
  * **실제 구현:** `g2b_scraper_service.py`가 실제 API 키값에 맞게 매핑하도록 긴급 수정하여 데이터 누락 해결
* **관심 키워드 모델링 정교화**
  * **기획(ERD):** `USER_MATCH_RULES`에 포함/제외 키워드를 text로만 관리
  * **실제 구현:** 단일 대표 키워드 저장을 위한 `keyword` (VARCHAR) 컬럼 추가로 매칭 검색 속도 증대

### 5. 운영 및 디버깅 단계 추가 고도화
* **사용자 대시보드 통계 데이터 동적 연동**
  * **기획(PRD):** 단순 UI 형태로 목업 데이터 제공
  * **실제 구현:** 맞춤 공고 데이터 API를 연동하여 핵심 통계를 실시간으로 계산해 제공
* **비밀번호 재설정 기능 완전 자동화 신설**
  * **기획(PRD):** 초기 로그인 페이지에 비밀번호 찾기 기능 누락
  * **실제 구현:** 무작위 임시 비밀번호를 발급하여 암호화 후 즉시 이메일(SMTP)로 발송하는 전체 사이클 신설
* **AI 요약 모델 동적 탐색 및 타임아웃 안정화**
  * **발생 원인:** 404 모델 미발견 오류 및 긴 요약 응답 시간으로 인한 타임아웃
  * **실제 구현:** `genai.list_models()`를 통해 최적의 모델(flash)을 동적 탐색하고 타임아웃을 60초로 연장
* **UI/UX 직관성(가시성) 개선 및 캘린더 고도화**
  * **실제 구현:** 박스형 패널 구조와 그림자, 캘린더 그리드 구분선 등 가시성 향상
* **이메일 스케줄러 시간대(Timezone) 고정**
  * **발생 원인:** 서버의 클라우드 환경(UTC)으로 인한 오전 9시 정각 발송 스케줄 어긋남
  * **실제 구현:** `APScheduler`의 시간대를 명시적으로 한국 시간(`Asia/Seoul`)으로 고정하여 정확성 보장

## 결과

- **장점**: 초기 기획의 단순함을 극복하고, B2B SaaS로서 실제 서비스 운영이 가능한 수준의 완성도, 보안, 관리 편의성을 확보했습니다. 프로젝트 발표 시 **실무 중심의 애자일 대처 능력**을 강력하게 어필할 수 있습니다.
- **단점**: 소셜 로그인, 권한 처리, 관리자 대시보드 추가 등으로 프론트엔드와 백엔드 모두 코드 복잡도가 초기에 비해 상승했습니다.
- **후속 작업**: 추가된 로직들에 대한 테스트 코드 작성 및 각 소셜(Kakao/Google) OAuth 콜백 예외 상황에 대한 에러 핸들링 보강이 필요합니다.
