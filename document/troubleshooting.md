# 🛠️ 트러블슈팅 및 문제 해결 사례 (발표용)

본 문서는 프로젝트 개발 및 배포 과정에서 발생했던 주요 기술적 이슈들과 이를 어떻게 분석하고 해결했는지에 대한 트러블슈팅 사례를 정리한 문서입니다. 발표 시 **[문제 해결 경험]** 또는 **[개발 주안점]** 슬라이드에 활용하기 좋습니다.

---

## 1. 공공데이터 API 응답 스펙 불일치 문제 (데이터 수집 파트)

* **상황 (Issue):** 나라장터 OpenAPI를 연동하여 수집(Scraping)을 진행할 때, 공고의 '마감일' 데이터가 계속해서 `NULL`로 수집되거나 화면에 노출되지 않는 현상 발생.
* **원인 분석:** 조달청 공식 오픈 API 가이드 문서와 ERD 상으로는 마감일 키값이 `bidEndDt`일 것으로 예상했으나, **실제 API 응답의 JSON Payload를 분석해본 결과 `bidClseDt`라는 다른 키값으로 데이터가 넘어오고 있었음.**
* **해결 (Solution):** 백엔드 데이터 수집 스크래퍼(`g2b_scraper_service.py`)의 데이터 매핑 로직을 즉각 수정하여 `bidEndDt` 대신 `bidClseDt`를 파싱하도록 변경. 이후 모든 공고의 마감일 데이터가 정상적으로 DB에 적재되고 화면에 노출됨.
* **배운 점:** 외부 공공 API 연동 시, 공식 문서에만 의존하지 않고 실제 반환되는 Raw 데이터를 직접 디버깅하고 검증하는 과정이 필수적임을 깨달음.

## 2. 보안 로직 누락에 따른 권한 탈취/잠금 버그 (백엔드 파트)

* **상황 (Issue):** 
  1. 관리자가 특정 악성 유저의 상태를 '정지(blocked)'로 변경했음에도 불구하고, 해당 유저가 카카오/구글 소셜 로그인을 통해 버젓이 서비스에 접속할 수 있는 문제 발견.
  2. 최고 관리자가 실수로 본인의 권한을 일반 유저(USER)로 내리거나 본인을 정지시켜 영영 관리자 페이지에 접속할 수 없게 되는 치명적인 조작 실수 가능성 발견.
* **해결 (Solution):** 
  * **인증(Auth) 로직 보완:** 소셜 로그인 성공 직후 JWT 토큰을 발급하기 직전, DB에서 유저의 상태(`status`)를 조회하여 'blocked'인 경우 에러 메시지와 함께 403 Forbidden을 반환하도록 차단 로직 추가.
  * **어드민(Admin) 로직 방어:** `admin_routes.py`에서 권한 변경, 상태 변경, 회원 삭제 요청이 들어올 때, 타겟 `user_id`와 요청을 보낸 관리자 본인의 ID(`request.user['sub']`)를 비교하여 같을 경우 **"자기 자신은 변경/삭제할 수 없습니다"** 라며 요청을 반려하는 방어 코드 작성.
* **어필 포인트:** 시스템의 겉보기 완성도뿐만 아니라, 실제 운영 시 발생할 수 있는 '엣지 케이스(Edge Case)'와 '보안 취약점'을 선제적으로 찾아내고 탄탄하게 방어해낸 꼼꼼함을 어필 가능.

## 3. 프론트엔드 비동기(Promise) 에러 핸들링 누락 (프론트엔드 파트)

* **상황 (Issue):** 관리자 대시보드에서 잘못된 조작(예: 자기 자신 삭제 시도)을 했을 때 백엔드에서는 정상적으로 400 Bad Request 에러와 친절한 안내 메시지를 내려주었지만, 화면에는 아무런 경고창이 뜨지 않고 무반응인 현상.
* **원인 분석:** Axios 통신 시 HTTP 상태 코드가 200번대가 아니면 `.catch()` 블록으로 빠지는데, 기존 코드에서는 `catch(err => console.error(err))`로 단순히 브라우저 개발자 도구 콘솔에만 에러를 찍고 화면(Alert)으로 메시지를 띄워주는 로직이 누락되어 있었음.
* **해결 (Solution):** 모든 API 호출의 `catch` 블록을 리팩토링하여 `alert(err.response?.data?.message || "처리 중 오류가 발생했습니다.")` 형태로 백엔드의 에러 메시지를 사용자가 직접 인지할 수 있도록 UI/UX를 개선.

## 4. React 컴포넌트 구조 꼬임으로 인한 빌드(Build) 실패 (인프라 파트)

* **상황 (Issue):** 관리자 화면에 '회원 관리' 탭 기능을 덧붙인 후 클라우드타입에 배포를 시도했으나, 로컬 개발 서버에서는 보이던 화면이 배포 단계에서 `npm run build` 스크립트 에러(Syntax Error)를 뿜으며 중단됨.
* **원인 분석:** JSX 코드에서 복잡한 조건부 렌더링(삼항 연산자)을 수정하던 중 닫는 태그(`</div>` 및 `</>`)가 일부 누락되거나 잘못된 위치에 배치되어 컴포넌트 트리가 완전히 붕괴된 상태였음.
* **해결 (Solution):** 터미널의 빌드 에러 로그(Line number)를 추적하여 구조가 꼬인 부분을 정확히 식별하고, 손상된 Layout 태그들을 원상 복구하여 빌드를 성공시킴. 
* **배운 점:** 복잡한 UI 뷰를 하나의 거대한 컴포넌트로 짜기보다는, `UserManagementTable`, `ServiceStatusCard` 등과 같이 작은 컴포넌트로 분리(Componentization)하여 유지보수성을 높이는 것이 중요함을 깨달음.

## 5. 서버 환경 Timezone 설정 누락으로 인한 예약 작업(이메일 발송) 지연 버그 (백엔드 파트)

* **상황 (Issue):** 매일 오전 9시에 사용자 맞춤 공고 메일이 자동 발송되도록 APScheduler로 예약 작업(`hour=9`)을 설정해 두었으나, 클라우드타입(Cloudtype)에 배포한 이후 지정된 시간에 메일이 발송되지 않는 현상 발생.
* **원인 분석:** 로컬 환경과 달리 클라우드 배포 서버(OS)의 기본 시간대(Timezone)가 한국 시간(KST)이 아닌 협정 세계시(UTC)로 설정되어 있었음. 스케줄러에 별도의 Timezone을 명시하지 않아, 서버가 `hour=9`를 UTC 기준 오전 9시(한국 시간 기준 오후 6시)로 인식하여 실제 메일 발송이 9시간 지연 대기되는 것이 근본 원인.
* **해결 (Solution):** `scheduler.py`의 `BackgroundScheduler` 초기화 코드에 `timezone="Asia/Seoul"` 파라미터를 명시적으로 추가하여, 서버 운영체제의 기본 시간 설정에 의존하지 않고 항상 무조건 한국 시간(KST)을 기준으로 예약 작업이 실행되도록 수정함.
* **배운 점:** 로컬 개발 환경과 실제 배포 환경(클라우드, 서버 등)은 인프라 설정(특히 시간대)이 다를 수 있다는 점을 인지함. 향후 날짜나 시간에 민감한 스케줄링 작업을 구현할 때는 환경에 의존하지 않고 코드 레벨에서 명시적으로 Timezone을 강제 지정하는 것이 안전함을 배움.

## 6. AI 요약 기능 타임아웃 및 모델 미지원 404 에러 (AI 연동 파트)

* **상황 (Issue):** 
  1. 공고 상세 페이지에서 'AI 핵심 3줄 요약 받기' 기능 호출 시, 일정 시간이 지나면 프론트엔드에서 'AI 요약 요청 중 오류가 발생했습니다.' 라며 연결을 끊어버리는 타임아웃 현상 발생.
  2. 타임아웃 해결 후 배포 시, `404 models/gemini-2.5-flash is not found` 와 같은 에러가 반환되며 요약 기능이 완전히 동작하지 않는 현상 발생.
* **원인 분석:** 
  1. **타임아웃:** 생성형 AI 모델이 방대한 공고 원문을 분석하고 요약하는 데 통상 10~15초 이상이 소요되나, 프론트엔드 Axios 인스턴스의 글로벌 기본 타임아웃이 10초로 짧게 설정되어 있어 AI 응답이 오기 전에 연결을 강제 종료해버림.
  2. **404 모델 에러:** 구글 제미나이(Gemini) API의 버전이나, 서버 환경에 주입된 API 키의 권한 수준에 따라 특정 하드코딩된 모델명(`gemini-2.5-flash`, `gemini-pro` 등)에 대한 접근 권한이 없거나 지원이 중단되어 API 호출이 거부됨.
* **해결 (Solution):** 
  * **타임아웃 연장:** 프론트엔드 `NoticeDetail.jsx`에서 AI 요약 API를 호출하는 부분만 국소적으로 Axios `timeout` 옵션을 60000(60초)으로 넉넉하게 상향 조정하여 충분한 대기 시간을 확보.
  * **동적 모델 탐색 로직 적용:** 백엔드 `ai_service.py`에서 모델명을 하드코딩하지 않고, 호출 시 `genai.list_models()`를 통해 현재 API 키가 접근 허용된 전체 모델 목록을 먼저 조회하도록 변경. 그 중 텍스트 생성(`generateContent`)을 지원하는 제미나이 모델(처리 속도가 빠른 `flash` 계열 최우선)을 동적으로 선택해 요청하도록 유연하게 로직 개편.
* **배운 점:** 외부 의존성(Third-party API)을 연동할 때는 외부 환경이나 권한 정책 변화에 의해 스펙(버전, 모델명 등)이 언제든 바뀔 수 있으므로, 하드코딩을 피하고 API가 제공하는 메타데이터 조회를 활용하는 유연하고 방어적인 코드 작성이 서비스 중단을 막는 핵심임을 체감함.
