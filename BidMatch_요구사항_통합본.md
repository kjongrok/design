> 통합 기준: 상세 요구사항 문서의 번호 체계와 설명을 중심으로 유지하고, 나중에 작성된 `BidMatch_요구사항_ID_최종.md`의 페이지 ID와 요구기능 ID를 각 상세 섹션에 반영했다.
> 최종본에만 있는 추가 화면은 관련 상세 섹션 안 또는 페이지별 요구사항 끝부분에 별도 번호로 배치했다.
## 1. 문서 목적

본 문서는 BidMatch 프로젝트의 화면별 상세 요구사항을 정리한 문서이다.  
요구사항은 **페이지 단위로 그룹화**하여 정리하였으며, AI 요약, 임베딩 매칭, RAG, 자격 자가진단처럼 AI 또는 RAG 기술이 필요한 기능은 별도로 **[AI]** 표시를 추가하였다.

---

## 2. 프로젝트 개요

### 2.1 서비스명

**BidMatch**

### 2.2 서비스 목적

BidMatch는 나라장터 입찰 공고를 수집하고, 사용자의 관심 조건과 기업 정보를 기반으로 적합한 공고를 추천하는 서비스이다.

사용자는 공고를 직접 반복 검색하지 않아도 대시보드에서 추천 공고, 마감 임박 공고, 최근 알림을 확인할 수 있다. 또한 공고 상세 화면에서 AI 요약 결과와 자격 자가진단 결과를 확인하여 입찰 가능성을 빠르게 판단할 수 있다.

### 2.3 핵심 목표

- 나라장터 공고 데이터 자동 수집
- 사용자 관심 조건 기반 공고 매칭
- 임베딩 유사도 기반 매칭 점수 산정 **[AI]**
- AI 요약을 통한 공고 이해 시간 단축 **[AI]**
- 공고 원문/첨부파일 기반 RAG 질의응답 또는 근거 추출 **[AI/RAG]**
- 기업 정보 기반 자격 자가진단 제공 **[AI/Rule]**
- 이메일/알림 전송 이력 관리
- 관리자 검수 및 로그 조회 기능 제공
- 매칭 결과에 대한 O/X 피드백 수집

| 사용자 유형 | 설명 | 주요 기능 |
|---|---|---|
| 일반 회원 | 기업 인증 전 사용자 | 회원가입, 로그인, 공고 검색, 관심 공고 등록, 기업 인증 신청 |
| 기업 회원 | 기업 정보를 등록하고 승인된 사용자 | 맞춤 공고 추천, AI 요약 확인, 자격 자가진단, 관심 조건 관리, 자사 실적 관리 |
| 관리자 | 플랫폼 운영 및 데이터 관리 담당자 | 회원/기업 승인, 자격진단 검수, 수집 로그, AI 로그, 전송 로그 조회 |

## 5.2 임베딩 유사도 기반 매칭 **[AI]**

### 기능 개요

BidMatch의 매칭 점수는 단순 키워드 포함 여부가 아니라, **사용자 관심 조건 텍스트와 공고 텍스트의 임베딩 벡터 유사도**를 기준으로 산정한다.

사용자가 등록한 관심 조건을 하나의 문장 또는 문서 형태로 구성하고, 나라장터 공고 정보도 검색 가능한 문서 형태로 구성한 뒤 각각 임베딩한다. 이후 두 벡터 간 코사인 유사도를 계산하여 공고 적합도를 점수화한다.

### 매칭 흐름

1. 사용자가 관심 조건을 등록한다.
2. 관심 조건 텍스트를 생성한다.
3. 관심 조건 텍스트를 임베딩 벡터로 변환한다. **[AI]**
4. 나라장터 공고 데이터를 수집한다.
5. 공고 제목, 내용, 품목, 기관, 지역, 예산, 자격조건 등을 결합하여 공고 텍스트를 생성한다.
6. 공고 텍스트를 임베딩 벡터로 변환한다. **[AI]**
7. 관심 조건 벡터와 공고 벡터 간 코사인 유사도를 계산한다. **[AI]**
8. 유사도를 0~100점 기준 매칭 점수로 변환한다.
9. 점수가 높은 공고를 추천 공고로 표시한다.
10. 사용자의 O/X 피드백을 저장하여 향후 매칭 개선에 활용한다.

### 사용자 관심 조건 임베딩 텍스트 예시

```text
관심 분야는 정보시스템 구축, 소프트웨어 개발, 시스템 유지보수이다.
제외 분야는 청소, 인쇄, 단순 물품 구매이다.
관심 지역은 서울, 경기이다.
선호 기관은 공공기관과 지자체이다.
예산은 1천만 원 이상 공고를 선호한다.
보유 면허는 소프트웨어사업자, 정보통신공사업이다.
```

### 공고 임베딩 텍스트 예시

```text
공고명: 공공기관 정보시스템 유지보수 용역
수요기관: OO시청
공고내용: 행정 시스템 유지보수 및 기능 개선 사업
품목명: 소프트웨어 개발 서비스
지역: 경기
예산: 50000000
입찰 참가 자격: 소프트웨어사업자 등록 업체
첨부 요약: 유지보수 인력 투입 및 장애 대응 필요
```

### 매칭 점수 산정 방식

```text
cosine_similarity = dot(user_condition_embedding, bid_notice_embedding)
                    / (norm(user_condition_embedding) * norm(bid_notice_embedding))

matching_score = cosine_similarity * 100
```

### 점수 구간 예시

| 매칭 점수 | 판단 |
|---:|---|
| 85점 이상 | 매우 적합 |
| 70점 이상 | 적합 |
| 55점 이상 | 검토 필요 |
| 55점 미만 | 추천 제외 또는 낮은 우선순위 |

### 규칙 필터와의 관계

임베딩 유사도는 공고 내용의 의미적 유사도를 판단하는 기준으로 사용한다.  
단, 명확히 제외해야 하는 조건은 규칙 필터로 처리한다.

| 조건 | 처리 방식 |
|---|---|
| 제외 키워드 포함 | 추천 제외 또는 강한 감점 |
| 마감 완료 공고 | 추천 제외 |
| 지역 제한 불일치 | 감점 또는 자격진단에서 주의 표시 |
| 필수 면허 불일치 | 자격진단에서 불가능 표시 |
| 예산 범위 불일치 | 감점 또는 낮은 우선순위 처리 |

| 기업회원 | 대표자명과 별도로 서비스 담당자 이름을 저장한다. |
| 기업 인증 | 사업자등록증 파일, 신청 상태, 승인·반려 사유 및 처리 일자를 관리한다. |
| 고객센터 | 공지사항과 1:1 문의·답변·처리 상태를 저장한다. |
| 관심 조건 | 여러 조건 중 현재 매칭에 사용하는 단일 활성 조건을 식별한다. |
| 사전규격 | 사전규격 정보, 관심 등록 및 정식 공고 연결 정보를 저장한다. |
| 개찰결과 | 기초금액, 예정가격, 낙찰금액·낙찰률·낙찰업체와 참여업체별 투찰정보를 저장한다. |

## 6.3 DB 설계 기준

- 공고번호는 중복 저장되지 않도록 유니크 처리한다.
- 회원과 기업 정보는 분리한다.
- 일반 회원과 기업 회원 상태를 구분한다.
- 관심 조건은 사용자별 다건 등록 가능해야 한다.
- 관심 조건과 공고 임베딩 벡터를 저장할 수 있어야 한다.
- RAG를 위해 공고 원문/첨부파일 청크와 청크별 임베딩을 저장할 수 있어야 한다.
- 매칭 결과는 사용자, 조건, 공고 기준으로 저장한다.
- 임베딩 유사도 원본 값과 0~100점 변환 점수를 함께 저장한다.
- AI 요약 결과와 자격진단 결과는 공고별 또는 사용자-공고별로 저장한다.
- 로그성 데이터는 조회 성능을 고려해 인덱스를 설계한다.

## 12.3 작성 시 주의사항

- 요구사항 분석표의 ID에는 페이지 ID가 아니라 요구기능 ID를 적는다.
- 스토리보드 상단에는 페이지 ID를 적는다.
- 스토리보드 내부의 `A1`, `D1`, `V1` 표시는 기능 ID와 별도로 유지한다.
- 고객센터 화면 ID는 현재 작성된 `CS-001`, `CS-002`를 사용한다.
- 자격진단 검수 큐와 관련된 화면 및 요구사항은 추가하지 않는다.
