## 1. 프로젝트 개요

**BidMatch**는 나라장터(G2B)에 게시되는 공공 입찰 관련 정보(입찰공고, 사전규격, 개찰결과·낙찰정보)를 자동 수집하고, 기업의 업종·보유 면허·실적·관심 키워드를 기반으로 **AI가 적합한 공고를 추천·분석**해주는 서비스입니다.

### 1.1 문제 정의
- 나라장터에는 하루 수백~수천 건의 공고가 등록되지만, 키워드 단순매칭 검색은 재현율은 높으나 정밀도가 낮아 관련 없는 공고까지 다수 노출됨
- 담당자는 공고 원문을 일일이 읽고 자격 요건을 대조해야 하는 시간 비용 문제를 겪음
- "우리 회사가 이 공고에 지원할 자격이 되는가"를 자동으로 알려주는 서비스가 시장에 부재함

### 1.2 차별화 전략 (팀 확정)
벤치마킹 결과, 낙찰가 예측(디마툴즈)·경쟁사 분석(비딩톰)·사업자 검색(G2BMarket) 등은 이미 시장에 강한 경쟁자가 존재합니다. **BidMatch는 이런 영역에서 정면 경쟁하지 않고, 아래 두 가지에 집중합니다.**
1. **자격 자가진단 ⭐** — 시장에 없는 기능. "낼 수 있는 공고만 걸러준다"는 명확한 가치
2. **나라장터 서비스의 원스톱 편의성** — 입찰공고뿐 아니라 사전규격·개찰결과·입찰제안서 지원까지, 나라장터에 흩어진 여러 화면을 BidMatch 한 곳에서 편리하게 이용

> 낙찰가 예측/경쟁률 분석/발주기관 트렌드 분석 등은 본 프로젝트 범위에서 **명시적으로 제외(Won't Have)** 합니다. (8절 참고)

## 3. 타겟 사용자 및 회원 등급 (신설)

기업 등록 여부에 따라 **2단계 회원 등급**을 둡니다. (예비 창업자/사업자도 서비스를 먼저 체험할 수 있도록)

| 등급 | 대상 | 이용 가능 기능 |
|---|---|---|
| **일반회원** | 예비 사업자, 사업 구상 단계의 개인 등 | 공고 열람/검색, 관심공고 등록, 캘린더, AI 공고 요약(일반 열람용) — "이런 사업을 하면 어떤 공고가 있을까" 탐색 목적 |
| **기업회원** | 사업자등록증을 보유한 실제 기업 담당자 | 일반회원 기능 + **자격 자가진단**, AI 기업 매칭 스코어, 실적 관리, 맞춤 알림 |

**전환 흐름**: 일반회원으로 가입 → 마이페이지에서 사업자등록증 등록(OCR) → 기업정보 승인 후 **기업회원으로 자동 전환** → 자격진단/매칭 기능 활성화

이 구조는 온보딩 장벽을 낮추는 동시에, 회원가입 직후 이탈을 줄이는 효과를 기대할 수 있습니다.

## 7. AI 핵심 기능 상세 설계

### 7.1 자격 자가진단 ⭐ (하이브리드 로직 — 팀 논의 반영)

**설계 방향**: 규칙 기반과 LLM 판단을 절반씩 섞고, 신뢰도 확보를 위해 관리자 검수 루프를 둡니다.

```
[1] 공고 원문 파싱
    LLM(Qwen3-8B)이 참가자격 조항에서 구조화된 요건 추출
    (면허 종류 / 지역 제한 / 실적 조건) + 근거 조항(원문 인용) 함께 저장
        ↓
[2] 규칙 기반 1차 대조
    추출된 요건 vs 기업 프로필(면허/지역/실적)을 규칙(rule)으로 매칭
    → 예비 판정: 가능 / 불가 / 애매(조건 불명확)
        ↓
[3] 관리자 검수 큐 (신규 요건 유형 · 신뢰도 낮은 케이스만)
    AI가 추출한 "근거 조항 + 예비 판정"을 관리자 화면에 노출
    관리자가 동의하면 → 최종 승인 상태로 전환, 해당 요건 유형은
    이후 규칙 마스터 데이터에 반영되어 자동화 정확도가 점진적으로 향상
    (모든 진단을 매번 관리자가 검수하는 것이 아니라, 신규/저신뢰 케이스만 큐에 쌓는 구조 — 실시간 응답성 유지)
        ↓
[4] 사용자 화면 노출
    "입찰 가능" / "참가 불가(사유: OO면허 필요)" + 근거 조항 인용
    + "본 결과는 참고용이며, 최종 확인은 공고 원문 대조가 필요합니다" 문구 고정 노출
```

> 💡 이 구조는 처음엔 관리자 검수 비중이 높다가, 규칙 마스터 데이터가 쌓일수록 자동 판정 비중이 늘어나는 **점진적 신뢰 구축형 구조**입니다. 4주 프로젝트에서는 "검수 큐 UI + 승인 시 마스터 데이터 반영"까지만 MVP로 구현하고, 완전 자동화는 이후 과제로 남겨도 됩니다.

### 7.2 AI 의미 기반 검색 (Semantic Search)
- 공고 제목+본문+첨부파일 텍스트를 임베딩 → Chroma(임베디드 벡터DB)에 저장, 키워드 검색과 하이브리드로 병행

### 7.3 AI 공고 요약/분석
- Spring Boot(REST API)가 FastAPI 엔드포인트 호출 → Qwen3-8B(sLLM)로 요약/구조화
- 첨부파일 텍스트 추출이 선행되어야 함 (HWP 파싱 난이도 — 14절 리스크 참고)

### 7.4 사업자등록증 OCR (YOLOv8-seg 활용)
- YOLOv8-seg로 사업자등록증 이미지 내 텍스트 영역(상호/사업자번호/대표자 등)을 우선 분할(segmentation) → 분할된 영역에 OCR 적용하는 2단계 파이프라인으로 추정됩니다.
- ⚠️ 정확한 활용 목적은 15절 열린 질문에서 재확인이 필요합니다.

## 8. 기술 스택 (팀 확정, 변경 가능)

| 레이어 | 기술 |
|---|---|
| 웹/리버스 프록시 | Nginx → Spring Boot 내장 Tomcat |
| 백엔드(메인 API) | Spring Boot 3.x, JDK 17, Gradle → bootJar로 OCI 배포 |
| 백엔드(AI 서빙) | FastAPI (Python 3.12) — Spring Boot이 REST로 호출하는 내부 엔드포인트 |
| DL/LLM | YOLOv8-seg(이미지 분할), Qwen3-8B sLLM(HuggingFace 연동) |
| 프론트엔드 | React + Axios |
| 스타일링 | Tailwind CSS |
| DB | PostgreSQL(RDB), Redis(Spring Session/분석 캐시/챗봇 대화 메모리), Chroma(임베디드 벡터DB) |
| 배포 | OCI(Oracle Cloud Infrastructure), SSH 배포, dev/prod 환경 분리 |
| 아키텍처 | 모놀리식(Spring Boot 중심) + AI 서빙만 FastAPI로 분리 |
| 컨테이너 | Docker Compose (팀 논의에 따라 변경 가능) |

**아키텍처 흐름(요약)**
```
Client(React) → Nginx → Spring Boot(REST API, JDK17/Gradle)
                              ├─ PostgreSQL (정형 데이터: 회원/기업/공고)
                              ├─ Redis (세션/캐시/챗봇 메모리)
                              └─ FastAPI(Python 3.12)
                                     ├─ YOLOv8-seg (OCR 전처리)
                                     └─ Qwen3-8B (요약/자격진단/검색 임베딩)
                                            └─ Chroma (임베디드 벡터DB)
```

**참고 — 자체 sLLM 운영 관련 유의점**: Qwen3-8B를 자체 호스팅할 경우 GPU 인프라(OCI GPU 인스턴스) 확보와 추론 속도가 KPI(3초 이내 응답)에 직접 영향을 줍니다. 초반에는 배치 처리·캐싱(동일 공고 재분석 방지)으로 부하를 줄이는 것을 권장합니다. (14절 리스크 참고)

## 9. 팀 구성 및 R&R (확정 — 페이지 단위 배정)

팀은 총 **5명**이며(종록님/안민영님/혜라님/하영님/명일님), 기존에 제안했던 레이어별(FE/BE/AI) 페어 구조 대신 **페이지(화면) 단위로 2인씩 배정**하는 방식으로 확정했습니다. 페이지 단위 분담은 한 사람이 화면 하나를 프론트~백엔드까지 책임지고 끝까지 완성할 수 있다는 장점이 있어, 이 프로젝트처럼 4주 안에 여러 화면을 병렬로 완성해야 하는 일정에는 합리적인 선택입니다. 아래는 팀에서 공유한 배정을 표로 정리한 것입니다.

## 13. 열린 질문 & 추가 제안사항 (업데이트)

### 확정됨 ✅
- 차별화 전략: 자격 자가진단 + 나라장터 원스톱 편의성에 집중, 낙찰가예측/경쟁분석 등은 Won't Have
- 기술 스택: Nginx/Spring Boot/FastAPI/React/Tailwind/PostgreSQL/Redis/Chroma/OCI, 모놀리식
- 회원 등급: 일반회원(예비 사업자) ↔ 기업회원 전환 구조
- 자격 자가진단 로직: 규칙+LLM 하이브리드, 관리자 검수 큐를 통한 점진적 신뢰 구축
- Disclaimer(참고용 문구), 나라장터 단일 소스 한정
- 데모 계정 필요성 인지 (구체적 시나리오는 개발 중 확정)

### 여전히 확인 필요
1. **YOLOv8-seg의 정확한 활용 목적**: 사업자등록증 OCR 전처리(문서 영역 분할)로 추정했는데 맞는지, 혹은 다른 이미지 분석 용도(예: 공고 첨부 도면/이미지 분석)가 있는지 확인 필요
2. **관리자 검수 큐의 트리거 기준**: "신뢰도 낮은 케이스"를 어떻게 정량화할지(예: LLM confidence score, 미리 정의된 면허 종류 화이트리스트 존재 여부 등) — 페어 C(AI 담당)에서 초기 규칙 설계 필요
3. **일반회원의 AI 요약 열람 범위**: 일반회원도 AI 공고 요약을 볼 수 있게 할지, 아니면 열람 기능은 제한하고 검색/필터만 제공할지(과금/리소스 관점에서 논의 필요 — Qwen3-8B 자체 호스팅 비용 고려)
4. **사전규격/개찰결과 메뉴의 API 소스 확인**: 나라장터 Open API 중 사전규격·개찰결과 각각의 API가 별도로 존재하는지, 데이터 포맷이 입찰공고와 동일한지 사전 확인 필요
5. **사용자 테스트 참여자 섭외**: 5.5절 리스크에 명시한 대로, 팀원 및 타 팀 학습자 대상 협조 요청 예정
